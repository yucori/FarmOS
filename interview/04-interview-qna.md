# 면접 Q&A — 예상 질문 14개

> 각 질문에 대해 **30초 답변**과 **2분 답변** 두 버전을 준비해 두면 면접관 톤에 맞춰 대응 가능.
> 답변 끝에 `→` 로 표시한 것은 면접관이 후속으로 던질 가능성이 높은 follow-up.

## 카테고리 1: 기술 의사결정

### Q1. "팀원들은 LangChain으로 AI Agent를 만들었는데 왜 본인은 안 썼나요?"

**30초**: "LangChain은 Python 라이브러리이고, MCP는 Anthropic이 정의한 표준 프로토콜입니다. 한 층 위입니다. 제 모듈은 MCP로 10개 tool을 노출했고, 이 tool들은 `langchain-mcp-adapters`를 통해 팀의 LangChain Agent도 그대로 호출할 수 있습니다. 즉 안 쓴 게 아니라 **상위 호환 표준**으로 만든 것입니다."

**2분**: 위 30초 답변 + 다음 4가지 근거
1. 이미 `BaseLLMClient` 팩토리로 multi-provider 추상화 달성 → LangChain의 vendor abstraction 효익이 이미 확보됨. 또 한 겹 얹으면 추상화 중복
2. 한국어 하이브리드 검색·텍스트 dedup·멀티테넌트 필터·SSE 진행률 같은 도메인 특화 로직은 LangChain Retriever로 감싸도 어차피 직접 작성 필요
3. 발표 8일 전 시점에서 LangChain v0.3 마이그레이션 + 의존성 추가는 회귀 테스트 부담이 효익을 초과
4. MCP는 Claude Desktop·Cursor 등에서 즉시 시연 가능 — 발표 데모 측면에서 더 강력

→ 후속 예상:
- "그래도 팀 통일성은요?" → Q2
- "LangChain 자체는 잘 아세요?" → Q3
- "RAG 그냥 LangChain Chroma 쓰면 더 빠르잖아요" → Q4

---

### Q2. "팀이 LangChain 표준인데 본인만 다른 거면 통일성 문제 아닌가요?"

**답**: "팀 모듈은 multi-step reasoning이 필요한 워크플로(예: 다단계 의사결정)이고, 제 모듈은 단방향 데이터 처리 파이프라인입니다. layer가 다릅니다. **tool 표준화는 MCP로 통일**되어 있고, LangChain Agent도 MCP tool을 호출할 수 있어 **양립 가능**합니다. 통일성은 'tool 호출 인터페이스 표준화'에서 이미 확보됐습니다."

→ 후속: "정말 양립 가능한가요?" → "네, `langchain-mcp-adapters` 패키지가 그 역할입니다. Anthropic 공식 + LangChain 커뮤니티가 같이 만든 어댑터입니다."

---

### Q3. "LangChain 모르는 거 아니에요?"

**답**: "Chain·Agent·Retriever·Tool·Memory 핵심 개념과 LCEL, LangGraph 차이는 알고 있습니다. 다만 도입 결정은 '아는가'가 아니라 '필요한가'로 합니다. 가장 ROI 높은 1곳 — `_parse_json_response`의 3단 폴백을 `with_structured_output(Pydantic)`로 교체하는 것 — 은 발표 후 점진 도입 PoC로 진행할 계획입니다."

(가능하면 PoC를 면접 전에 실제로 돌려두고 "이 커밋에서 도입했고 50줄 줄였습니다"로 답하면 강력함)

---

### Q4. "RAG는 그냥 LangChain Chroma + Retriever 쓰면 되잖아요?"

**답**: "처음엔 그렇게 시작할 수 있습니다. 그런데 한국어 리뷰에서 순수 벡터 검색만으로는 짧은 리뷰('별로...')가 상위에 잘 올라옵니다. 그래서 **벡터 + 키워드 부스팅(0.3)** 하이브리드 + 시드 데이터의 **텍스트 dedup**이 필요했습니다. LangChain Retriever로 감싸도 이 로직은 그대로 작성해야 합니다. 결국 추상화 비용만 추가됩니다."

→ 후속: "키워드 부스팅 수치는 어떻게 정했어요?" → "0.3은 코사인 유사도 0~1 범위에서 '키워드 일치 시 한 단계(약 30%) 우선'을 의미하는 경험적 값입니다. A/B 테스트로 0.1·0.3·0.5를 비교했고 0.3이 가장 균형 잡혔습니다." (실제로 안 했다면 이렇게 답하지 말 것 — 거짓말은 즉시 들킴. 대신 "현재는 경험적 값이고 측정 기반 튜닝은 향후 개선 항목"이라고 정직하게)

---

### Q5. "LangFlow는 검토 안 했나요?"

**답**: "검토했습니다. **시각적 빌더로 비개발자가 플로우를 편집하는 게 명시 요구사항이 아니면** 효익 대비 비용이 큽니다. 별도 서버 + 별도 DB 운영 부담, 플로우 JSON의 코드리뷰 부재(시각적 diff 불가), 멀티테넌트 컨텍스트(JWT의 seller_id) 전파의 복잡성 때문입니다. PoC 환경 한정이면 유효하지만 프로덕션엔 비추천했습니다."

## 카테고리 2: 리뷰 자동화 분석

### Q6. "리뷰 분석 비용 최적화 어떻게 했나요?"

**답**: "두 가지로 비용을 1/(3N)로 줄였습니다.
1. **1콜 다중 분석**: 감성·키워드·요약을 별도 호출하지 않고 단일 프롬프트로 동시에 받습니다. 1콜 = 3분석 → 비용 1/3
2. **배치 병렬**: 50건씩 묶어 `asyncio.gather`로 모든 배치 동시 호출. 1만건 분석을 30초 내로 마무리 → 호출 수 1/N

LLM은 클라우드 API라 동시 호출 가능해 N이 커질수록 효과 큽니다."

→ 후속: "배치 사이즈 50은 어떻게 정했어요?" → "토큰 한도(일반적으로 32K context window)와 JSON 응답 안정성을 고려한 경험값입니다. 100건 이상이면 LLM이 일부 누락하는 경우가 있어 50을 상한으로 설정했습니다."

---

### Q7. "LLM이 JSON 안 주면 어떻게 처리해요?"

**답**: "3단 폴백 파싱을 적용했습니다.
1. `json.loads()` 직접 시도
2. 실패 시 `\`\`\`json...\`\`\`` 코드블록 추출 후 파싱
3. 그래도 실패 시 첫 `{` ~ 마지막 `}` 슬라이싱

3단계도 실패하면 `JSONDecodeError`로 재시도 루프에 진입합니다. 재시도는 최대 N회(기본 3회)이고, **HTTP 4xx(영구 오류)는 즉시 포기**, **429/5xx(일시 오류)만 지수 백오프 재시도**합니다.

→ 향후 개선: `with_structured_output(Pydantic)`으로 스키마를 강제하면 이 3단 폴백을 제거할 수 있습니다."

---

### Q8. "ChromaDB 임베딩 실패 처리는요?"

**답**: "**영벡터 합성 금지** 원칙입니다. 임베딩 API가 실패하면 0으로 채워서 저장하지 않고 예외를 그대로 호출자에게 전파합니다. 영벡터를 저장하면 나중에 그 리뷰가 모든 검색에서 동일한 거리값으로 나오면서 컬렉션이 영구 오염됩니다.

청크 단위 처리에서는 **해당 청크만 skip**하고 다음 청크는 계속 처리합니다. 즉 1만건 중 100건이 실패해도 9900건은 정상 저장됩니다."

---

### Q9. "임베딩 모델 바꿀 때 마이그레이션은요?"

**답**: "ChromaDB는 컬렉션의 차원이 고정이라 모델/차원을 바꾸면 같은 컬렉션에 섞을 수 없습니다. 그래서 **컬렉션 이름을 함께 바꾸는 규약**을 만들었습니다 — 이전 `reviews_bge_m3` → 현재 `reviews_voyage_v35`. 운영 절차는 별도 룬북(`docs/runbooks/review-embedding-migration.md`)으로 문서화했습니다.

블루-그린 방식으로: 신규 컬렉션 생성 → 백그라운드 재임베딩 → 검증 → 라우팅 전환 → 구 컬렉션 삭제."

---

### Q10. "FastAPI랑 MCP 둘 다 노출한 이유는요?"

**답**: "사용자 진입점이 다르기 때문입니다.
- **FastAPI**: 웹 프론트엔드(React)에서 호출
- **MCP**: Claude Desktop·Cursor 등 AI 에이전트가 직접 호출

두 진입점이 같은 ChromaDB·같은 LLM 클라이언트를 공유하도록 `review_singletons.py`로 인스턴스를 분리했습니다. 한 곳에서 임베딩하면 양쪽에서 즉시 검색 가능하고, **코드 중복 0**입니다.

이전엔 `mcp/tools.py`가 `api/`를 import하는 비정상적 의존이 있었는데, 싱글턴 분리로 바로잡았습니다."

## 카테고리 3: IoT 적재 Bridge

### Q11. "IoT 부분 어떻게 구현했나요?"

**답**: "외부 Relay 서버에서 AI Agent가 결정을 만들고, 그 결과를 FarmOS Postgres에 미러링하는 ETL Bridge입니다. 핵심 설계는 4가지입니다.

1. **이중 채널** — SSE 실시간 + HTTP backfill. 재기동 동안 누락 방지
2. **멱등 UPSERT** — `INSERT ... ON CONFLICT (id) DO NOTHING`. 중복 무해
3. **복합 cursor pagination** — `(timestamp, id)`로 동일 timestamp 다중 행 누락 방지
4. **Exponential backoff** — 1→60s, Relay 다운 시에도 graceful

집계는 시간/일별로 자동 bump하고, `avg_duration_ms`는 null-duration 편향을 제거하기 위해 `duration_sum/duration_count`를 직접 누적합니다."

---

### Q12. "왜 결정 로직을 FarmOS 안에 두지 않았나요?"

**답**: "현재는 **책임 경계를 의도적으로 분리**했습니다. Relay는 센서 데이터에 가까이 있고 결정은 거기서 합니다. FarmOS는 결정 결과를 보존·분석·UI 노출에 집중합니다.

이 분리의 장점은 ① Relay 패치 미적용 환경에서도 FarmOS는 빈 결과로 안전하게 동작 ② Relay 결정 로직 변경이 FarmOS 배포에 영향 없음 ③ 마이크로서비스 분리로 장애 격리.

향후 결정 엔진을 FarmOS로 흡수하는 옵션은 검토 대상이지만, 그건 PRD 신규 작성 수준의 큰 변경이라 현재 범위에 포함하지 않았습니다."

---

### Q13. "메시지 큐(Kafka) 안 쓰고 SSE+HTTP만으로 괜찮나요?"

**답**: "현재 트래픽 규모(decision/sec ~ 단위)에선 충분합니다. SSE 실시간 + HTTP backfill + 멱등 UPSERT 조합으로 **at-least-once 보장**을 메시지 큐 없이 달성했습니다.

다만 한계는 인지하고 있습니다 ① Bridge는 단일 프로세스라 트래픽 폭증 시 병목 ② DLQ(Dead Letter Queue) 미구현으로 잘못된 payload는 WARN 로그만 남고 버려짐 ③ Relay 의존도 100%.

트래픽이 10배 이상 증가하면 Kafka/Redis Stream 도입을 검토할 것입니다. 현 시점에선 YAGNI(필요해질 때 도입) 원칙을 따랐습니다."

## 카테고리 4: 종합/소프트

### Q14. "이 프로젝트에서 가장 어려웠던 점은요?"

**답**(예시 — 본인 경험으로 교체): "팀이 LangChain 표준으로 가는 흐름에서 **다른 표준(MCP)을 선택하는 결정**이 가장 어려웠습니다. 동조하면 편하지만, 제 모듈의 워크플로(단방향 데이터 처리)에는 LangChain Agent의 multi-step reasoning이 과잉이라고 판단했습니다.

이를 '팀과 다른 길'이 아니라 '**상위 호환 표준 인터페이스 제공자**'로 포지셔닝했고, `langchain-mcp-adapters`로 양립 가능성도 확보했습니다.

기술적으로는 ChromaDB 임베딩 모델 마이그레이션이 어려웠습니다. 차원이 바뀌면 컬렉션 호환이 안 돼서 컬렉션 이름 규약 + 블루-그린 마이그레이션 룬북을 직접 만들어야 했습니다."
